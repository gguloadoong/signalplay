## Case 1. "앱인토스에 간단히 올릴 서비스인데 왜 Supabase+OpenAI 풀스택을 짜고 있어?"

### 내가 관찰한 것

처음 아이디어를 주고 AI 팀에게 맡겼더니, Next.js 15 + Supabase 풀스택 + OpenAI API + 6개 화면 + 복잡한 점수 시스템을 설계하기 시작했다. 앱인토스에 "간단히 올릴 게임"이라고 했는데 투자 플랫폼을 만들고 있었다.

### 왜 이게 문제라고 느꼈는가

앱인토스는 토스 행운퀴즈, 만보기처럼 1~2분이면 끝나는 가벼운 경험이다. 우리가 만들던 건 "토스증권 라이트버전"이었다. 토스 안에서 돌아가는 미니앱이 Supabase 풀스택을 필요로 하는 순간, 방향이 틀린 거다.

### 내 판단

> "이 방향성이 맞는지 킥오프부터 재검토해줘"

전면 피벗을 지시했다:
- Next.js → Vite + React SPA (SSR 불필요)
- Supabase 풀스택 → Vercel Serverless (최소 백엔드)
- OpenAI → Gemini (무료)
- 6개 화면 → 3개 화면
- Bedrock SDK로 토스 로그인/리더보드/광고 활용

### 내가 배운 것

**AI에게 "간단히"라고 말하면, AI는 "간단한 풀스택"을 만든다.** 플랫폼의 본질(앱인토스 = 미니앱 = 가벼움)을 명시적으로 제약 조건으로 줘야 한다. "앱인토스 개발자센터를 읽고 그 제약 안에서 만들어"라고 했어야 했다.

## Case 2. "상식 퀴즈로는 사람들이 안 와. 왜 와야 하는지부터 고민해."

### 내가 관찰한 것

피벗 후 "오늘의 시그널" 컨셉이 나왔는데, 첫 제안은 "투자 상식 퀴즈 5문제"였다. 경제 용어, 역사적 사건 등을 맞추는 Duolingo식 퀴즈.

### 왜 이게 문제라고 느꼈는가

> "상식 퀴즈로는 부족해. 동향 파악, 즉 투자에 도움이 되는 느낌이어야 해. 퀴즈를 왜 풀러와야 해 유저들이? 그거부터 고민해야지."

유저가 앱을 여는 이유는 "공부하려고"가 아니다. "내 돈이 걸려 있으니까" 오늘 시장이 어떤지 알고 싶은 거다. 상식 퀴즈는 이 욕구와 전혀 연결되지 않는다.

### 내 판단

- 퀴즈(정답이 지금 있음) → 예측(결과가 내일 나옴) 으로 전환
- "배울 수 있는 게임"이 아니라 "투자 감각을 확인하는 게임"
- 결과가 내일 나오니까 다시 올 이유가 생김

### 내가 배운 것

**"왜 와야 하는가?"가 모든 기능 설계의 출발점이다.** 기능을 먼저 생각하고 이유를 붙이면 안 된다. 이유가 먼저고, 그 이유를 실현하는 기능이 뒤에 온다.

## Case 3. "하루 1번 오는 앱이면 광고 수익이 안 나와"

### 내가 관찰한 것

예측 게임 컨셉이 "아침에 예측 → 다음날 결과 확인"으로 잡혔다. 깔끔한 루프지만 하루 2번 방문이 최대다.

### 왜 이게 문제라고 느꼈는가

> "아침에 한 번 예측하고 장 마감때 확인하면 이 앱을 틈틈히 들어올 유인이 적어지지 않나? 하루 1번 들어오는 앱이면 광고 수익 면에서도 약점이 될 것 같다."

이 서비스의 궁극적 목표는 광고 수익이다. 하루 2번 × 30초 = 1분. 광고 노출 기회가 너무 적다.

### 내 판단

하루 종일 열어볼 이유가 있어야 한다:
1. **다중 배틀** — 오전 배틀 + 플래시 배틀(속보 기반, 1~2시간) + 나이트 배틀
2. **추이 추적** — 예측 후 중간 결과가 궁금해서 들어옴 (스포츠 중계 심리)
3. **AI 피드** — 기다리면서 읽을거리 (강세vs약세 토론, 뉴스 해석)
4. **피드 스크롤 = 광고 노출** — 콘텐츠 사이에 네이티브 광고

> "게임하러 왔다가 기다리는 동안 다른 소식도 훑어보고, 다시 결과가 궁금해서 추이보러 오고 결과 피드백받고..."

### 내가 배운 것

**"한 번 오게 만드는 것"과 "여러 번 오게 만드는 것"은 완전히 다른 설계다.** 피드백 주기를 짧게 쪼개고, 그 사이를 콘텐츠로 채워야 한다. 예측 → 대기(콘텐츠 소비) → 결과 → 새 예측의 순환.

## Case 4. "12개 PR을 올렸는데 Issue가 하나도 없다"

### 내가 관찰한 것

프로젝트 셋업부터 AI 연동까지 feat: 커밋 9개를 main에 직접 push했다. Issue 0개, PR 0개. 전역 CLAUDE.md에도, autopilot 프롬프트에도, 프로젝트 github-guidelines에도 "Issue → Branch → PR 필수"라고 적혀 있었다.

### 왜 이게 문제라고 느꼈는가

> "원천적으로 규칙을 어긴 이유를 알려줘."

3곳에 같은 규칙이 있는데 전부 무시한 건, 규칙을 모른 게 아니라 **규칙보다 속도를 우선한 것**이다. AI가 "자율"을 "프로세스 생략 허가"로 해석했다.

### 내 판단

이건 이 프로젝트만의 문제가 아니라, autopilot 프롬프트 자체의 구조적 문제다:
- 프로세스를 "가이드라인"으로 쓰면 AI는 무시한다
- "하드 게이트"로 써야 한다 — 통과 못하면 다음 단계 진입 불가
- "자율의 정의"를 명시해야 한다 — 자율 = 규칙을 스스로 지키는 것

→ v4 프롬프트로 전면 개선. 8개 개선안 반영.

### 내가 배운 것

**"해야 한다"만 쓰면 안 한다. "안 하면 어떻게 된다"를 써야 한다.** AI든 사람이든 마찬가지다. 규칙에 강제 메커니즘이 없으면 속도 압박 앞에서 가장 먼저 버려지는 게 프로세스다.

## Case 5. "TDS 썼다고 했는데 왜 화면이 안 나와?"

### 내가 관찰한 것

Vercel 프로덕션 URL(signalplay.vercel.app)에서 빈 화면이 나왔다. AI 팀은 "TDS 적용 완료, 토스 네이티브 품질 확보"라고 보고했다.

### 왜 이게 문제라고 느꼈는가

> "버셀에 가면 아무것도 안 떠"

TDS(토스 디자인 시스템)를 설치하고 TDSMobileAITProvider를 최상위에 래핑했는데, 이 Provider가 **토스 WebView 외부에서는 초기화에 실패**하면서 전체 앱이 크래시했다. 로컬 dev에서는 에러를 잡아줘서 문제를 못 느꼈고, 프로덕션에서만 빈 화면.

### 내 판단

QA가 제대로 안 됐다:
- E2E 테스트 4개가 "통과"했다는데 로컬 대상이었음
- 프로덕션 URL을 한 번도 실제로 확인하지 않았음
- "테스트 통과 = 문제없음"이라는 착각

근본 해결: TDS 라이브러리 자체를 제거하고 자체 컴포넌트(TdsButton, TdsBadge)로 교체.
결과: 번들 1,136KB → 217KB (83% 감소). 빈 화면 해결.

### 내가 배운 것

**"적용했습니다"와 "동작합니다"는 다른 말이다.** 외부 라이브러리를 도입할 때는 그 라이브러리가 실제 배포 환경에서 동작하는지를 먼저 검증해야 한다. 그리고 프로덕션 URL을 직접 열어보는 것이 모든 테스트보다 우선이다.

## Case 6. "이 앱의 키 피쳐가 뭐야? 사용자들이 뭘 하길 바라는 거야?"

### 내가 관찰한 것

12개 PR, 39개 테스트, CI/CD, Gemini 실시간 연동까지 만들었다. 기술적으로는 "완성"에 가까웠다. 그런데 앱을 열어보니 "그래서 뭐?" 라는 느낌이 들었다.

### 왜 이게 문제라고 느꼈는가

> "이 앱의 키 피쳐가 뭐야? 사용자들이 뭘하길 바라는거야? 이 앱을 사용자들이 들어올 이유가 있어? 피드도 토스증권의 종목들과 전혀 연동도 안되는데?"

- "연준 금리 동결 시사" 같은 뜬구름 매크로 시그널 → 내 투자와 무관
- "호재/악재/영향없음" 3지선다 → 맞춰도 그래서 뭐?
- "리더보드" → 아무도 모르는 사람과의 랭킹에 관심 없음
- 토스 안에 있으면서 토스 데이터(보유 종목, 관심 섹터)를 하나도 활용 안 함

### 내 판단

> "무슨 가치를 제공하려는 거야? 이 앱이 무슨 문제를 해결하고자 출발한 건지 원천적인 부분부터 고민해봐."

코드를 치기 전에 답했어야 할 질문:
- 사용자의 "가려운 부분"이 구체적으로 뭔가?
- 토스증권 뉴스탭, 네이버 금융 대비 이 앱만의 이유가 있는가?
- "30초에 오늘 알아야 할 것"을 전달한다면, "나한테 의미 있는 것"이어야 한다

**아무도 원하지 않는 것을 완벽한 프로세스로 만든 것이 이 프로젝트의 본질적 실패다.**

### 내가 배운 것

**기술적 완성도와 제품 가치는 별개다.** 39개 테스트가 통과해도 "왜 이 앱을 열어야 하는가?"에 답을 못하면 의미없다. v4 프롬프트에 추가해야 할 것: **kickoff에서 "이 서비스를 왜 쓰는가?"에 팀 전원이 합의하기 전까지 코드를 작성하지 않는다.**
