## 1. 브랜치 전략

### 기본 구조

```
main          ← 프로덕션 (Vercel Production 배포)
  └─ feature/#이슈번호-설명    ← 새 기능 개발
  └─ fix/#이슈번호-설명        ← 버그 수정
  └─ chore/설명                ← 설정, 의존성 등
  └─ docs/설명                 ← 문서 작업
```

### 브랜치 규칙

- `main` 직접 push 금지 — `feat:`, `fix:` 작업은 반드시 PR을 통해 머지
- `refactor:`, `docs:`, `chore:` 작업은 main 직접 커밋 허용
- 브랜치 이름은 항상 소문자 + 하이픈 사용 (`feature/#12-signal-card-ui`)
- 작업 완료 후 머지된 브랜치는 즉시 삭제

## 2. 이슈 관리

### 이슈 생성 규칙

**`feat:` 또는 `fix:` 작업 시 PR 전 반드시 Issue를 먼저 생성한다.**

```bash
gh issue create \
  --title "feat: Daily Signal Game 예측 제출 UI 구현" \
  --body "$(cat <<'EOF'
## 작업 내용
- PredictionSlider 컴포넌트 구현
- 예측 제출 API 연동
- 제출 완료 애니메이션 추가

## 완료 조건
- [ ] 슬라이더 UI 구현
- [ ] 신뢰도 단계(1~5) 선택 가능
- [ ] 중복 제출 방지 처리
- [ ] 제출 후 확인 애니메이션 재생

---
🤖 Generated by Claude Code [claude-sonnet-4-6]
EOF
)" \
  --label "enhancement,ai-generated"
```

### 라벨 체계

**모든 Issue/PR에 두 종류의 라벨을 반드시 조합 부착:**

| 라벨 | 설명 | 적용 대상 |
|---|---|---|
| `ai-generated` | AI가 개입된 모든 작업 | 모든 AI 작업 |
| `enhancement` | 새 기능 추가 | feat: 작업 |
| `bug` | 버그 수정 | fix: 작업 |
| `documentation` | 문서 작업 | docs: 작업 |
| `refactor` | 리팩토링 | refactor: 작업 |
| `design` | 디자인/UI 작업 | UI 변경 |
| `infrastructure` | CI/CD, 설정 | chore: 작업 |
| `frontend` | 프론트엔드 관련 | src/app, src/components |
| `backend` | 백엔드/DB 관련 | supabase/, src/lib/api |

### 이슈 템플릿

`.github/ISSUE_TEMPLATE/` 디렉토리에 아래 2개 템플릿 유지:
- `feature_request.md` — 새 기능 요청
- `bug_report.md` — 버그 리포트

## 3. 커밋 규칙

### 커밋 메시지 형식

```
<말머리>: <한국어 설명> (#이슈번호)

[선택] 상세 설명
```

### 말머리 목록

| 말머리 | 용도 |
|---|---|
| `feat:` | 새 기능 추가 |
| `fix:` | 버그 수정 |
| `refactor:` | 기능 변화 없는 코드 개선 |
| `docs:` | 문서 수정 |
| `chore:` | 빌드 설정, 의존성, 기타 |
| `style:` | 코드 포맷팅, 세미콜론 등 |
| `test:` | 테스트 추가/수정 |
| `perf:` | 성능 개선 |

### 예시

```
feat: SignalCard 컴포넌트 구현 (#12)

- Framer Motion 진입 애니메이션 적용
- 섹터 칩, 임팩트 배지 포함
- Skeleton 로딩 상태 구현
```

### 원자적 커밋

- 기능 하나 완성 시마다 커밋 — 여러 변경사항을 하나로 뭉치지 않는다
- 커밋 크기 가이드: 파일 5개 이하, diff 200줄 이하 권장

## 4. PR 프로세스

### PR 생성 절차

```
1. gh issue create         → Issue 생성 (라벨: ai-generated + 작업 성격)
2. git checkout -b feature/#이슈번호-설명
3. 작업 + 커밋 + 푸시
4. gh pr create            → PR 생성 (본문에 "Closes #이슈번호" 포함)
```

### PR 템플릿

```markdown
## 변경 사항
-

## 구현 방법
-

## 테스트 방법
- [ ] 로컬에서 직접 확인
- [ ] Vitest 유닛 테스트 통과
- [ ] 주요 E2E 시나리오 확인

Closes #이슈번호

---
🤖 Generated by Claude Code [claude-sonnet-4-6]
```

### PR 크기 가이드

| 크기 | 파일 수 | 줄 수 | 권장 처리 |
|---|---|---|---|
| Small | ~5개 | ~100줄 | 당일 리뷰 목표 |
| Medium | ~15개 | ~400줄 | 1일 내 리뷰 |
| Large | 15개+ | 400줄+ | 작업 분리 검토 |

## 5. 코드 리뷰 (봇 + 사람)

### 자동 리뷰어 (4명)

PR 생성 시 아래 4개 봇이 자동으로 리뷰를 수행한다:

1. **Gemini Code Assist** — Google AI 코드 리뷰
2. **Qodo PR Agent** — 설명, 개선 제안, 보안 분석
3. **CodeRabbit** — 변경 요약, 코드 품질
4. **GitHub Copilot** — Copilot 리뷰

### 리뷰 대응 원칙

- 봇 리뷰가 완료되면 각 코멘트를 읽고 타당성을 판단한다
- **보안/버그 지적** → 즉시 코드 수정 후 re-push
- **성능 개선 제안** → 타당하면 반영, 오버엔지니어링이면 기각 근거 코멘트
- **스타일 제안** → 프로젝트 컨벤션 기준으로 채택/기각

### 리뷰 종합 (봇 4명 리뷰 완료 후)

4개 봇 리뷰가 모두 달리면 Claude Code가 "리뷰 종합" 코멘트를 PR에 작성한다:
- 동의 사항, 의견 충돌, 채택/기각 판단 정리
- 동시에 `@coderabbitai`와 `/ask` (Qodo)에 질문
- 봇 응답 확인 → "최종 판단" 코멘트 → 머지

## 6. CI/CD & 머지 규칙

### CI 파이프라인 (PR마다 자동 실행)

```
lint (ESLint) → typecheck (tsc --noEmit) → unit tests (Vitest) → Vercel Preview 배포
```

main 머지 시 추가: E2E tests (Playwright) → Vercel Production 배포

### 머지 규칙

- `feat:`, `fix:` 작업: PR을 통해서만 머지 (main 직접 push 금지)
- approve 불필요 — 봇 리뷰 확인 후 본인이 머지
- **CI 모두 통과 필수** (lint, typecheck, build, tests)
- Squash merge 사용 (커밋 히스토리 정리)

## 7. README 관리

### README.md 업데이트 기준

다음 변경 시 README.md 함께 수정:
- 새 환경변수 추가 시 `.env.example` + README 설정 섹션 업데이트
- 새 스크립트(`package.json`) 추가 시 README 명령어 목록 업데이트
- 아키텍처 변경 시 README 다이어그램 업데이트

### README 필수 섹션

1. 프로젝트 소개 (1~2문장)
2. 기술 스택
3. 빠른 시작 (로컬 개발 환경 설정)
4. 환경변수 목록 (`.env.example` 참조)
5. 팀 연락처

## 8. 보안 체크 (Pre-commit)

커밋 전 반드시 확인:

```bash
git status   # .env, *.pem, *.key 파일이 Staged에 없는지 확인
```

**절대 커밋 금지 파일:**
- `.env` (`.env.example`은 허용)
- `*.pem`, `*.key`, `*.p12`
- API 키가 하드코딩된 파일
- `supabase/.env`

Husky pre-commit hook으로 자동 감지 설정 권장:
```bash
npx husky add .husky/pre-commit "npx lint-staged"
```
