## ADR-005: 앱인토스 피벗 결정

- **날짜**: 2026-03-22
- **상태**: 승인됨
- **결정자**: PM 제이크, Strategist 노바, 전체 팀

### 맥락

초기 설계는 토스 미니앱을 배포 채널 중 하나로 설정했으나, 서비스 컨셉이 분산되어 있었다. 앱인토스를 주 플랫폼으로 삼고 서비스를 완전히 재설계하기로 결정했다.

핵심 변화:
- 투자 시그널 분석 플랫폼 → 경제 동향 파악 + 예측 게임 습관 앱
- 5개 화면(홈/게임/리더보드/인사이트/프로필) → 3개 화면(홈/결과/리더보드)
- 소셜 로그인(카카오/구글) → 토스 로그인 단독 (Bedrock SDK)
- 복잡한 배지/업적 시스템 → MVP에서 제외, Phase 3으로 이동

### 결정

서비스를 앱인토스 전용으로 피벗한다:
- **컨셉**: "오늘의 시그널" — 매일 3분, AI 뉴스 요약 3개 + 호재/악재/영향없음 예측
- **플랫폼**: 앱인토스(앱인토스) 단독. 별도 웹사이트 없음.
- **게임 루프**: 시그널 확인 → 예측 → 다음날 결과 확인 → 리더보드

### 근거

- **유통 채널 확보**: 앱인토스는 토스 앱의 2030 사용자에게 즉시 노출. 별도 앱 설치·가입 없음.
- **단순화**: 3개 화면, 1가지 게임 루프로 집중. MVP 출시 기간 단축.
- **규정 리스크 감소**: 암호화폐 제외, 투자 자문 명시 제외로 앱인토스 심사 통과 가능성 향상.
- **수익화 즉시 가능**: AdMob(Bedrock SDK) Day 1 광고 수익 가능.

### 결과

**장점:**
- MVP 출시 기간 대폭 단축 (4주 → 1주 목표)
- 토스 사용자 기반 즉시 활용
- 인프라 비용 최소화 (Gemini 무료 + Vercel + Supabase 소규모 스키마)

**단점/리스크:**
- 앱인토스 정책 변경 시 서비스 전체 영향
- Bedrock SDK 의존도 높음 — SDK 업데이트/변경 대응 필요
- 토스 앱 외 접근 불가 (독립 웹 서비스 없음)

**후속 조치:**
- ADR-001~004는 대체됨으로 표시
- 앱인토스 심사 기준 문서 확보 (제이크 담당)

## ADR-006: AI 모델 — Google Gemini API 선택

- **날짜**: 2026-03-22
- **상태**: 승인됨
- **결정자**: BE 볼트, PM 제이크

### 맥락

시그널 카드 생성에 AI API가 필요하다. OpenAI GPT-4o-mini를 사용했던 구버전 대비 비용과 무료 한도를 재검토.

### 결정

**Google Gemini API (gemini-2.5-flash)** 무료 티어 사용.

### 근거

- **무료 티어**: Gemini 무료 티어는 분당 15회 요청. 일 3개 시그널 생성 + 결과 평가(3건)에 충분. 월 비용 $0.
- **충분한 품질**: 80자 이내 뉴스 요약 + 3지선다 결과 평가 작업에 gemini-2.5-flash 성능 충분.
- **OpenAI 대비**: gpt-4o-mini는 월 $10~30 예상. MVP 단계에서 불필요한 비용.
- **확장성**: 트래픽 증가 시 유료 티어로 전환 용이.

### 결과

**장점:**
- MVP 비용 $0 (AI 부문)
- Google 인프라 안정성

**단점/리스크:**
- Gemini API 무료 티어 Rate Limit(분당 15회) — Cron Job 1회 실행으로 처리하므로 문제없음
- OpenAI 대비 한국어 뉘앙스 차이 가능 → 프롬프트 검토 필요

**후속 조치:**
- 시그널 품질 샘플링 (볼트) — 첫 2주 생성된 시그널 수동 검토

## ADR-007: 프론트엔드 프레임워크 — Vite+React SPA 전환

- **날짜**: 2026-03-22
- **상태**: 승인됨
- **결정자**: FE 블레이즈, PM 제이크

### 맥락

구버전은 Next.js 15 App Router를 사용했다. 앱인토스 WebView SPA 환경에서 Next.js의 SSR/RSC 기능이 실제로 필요한지 재검토.

### 결정

**Vite + React 18 SPA**로 전환.

### 근거

- **앱인토스 = WebView SPA**: 토스 앱이 WebView로 SPA를 로드. Next.js SSR/ISR/RSC는 이 환경에서 이점 없음. Vercel Edge에서 정적 파일로 서빙하면 충분.
- **Bedrock SDK 호환성**: Bedrock SDK는 브라우저 컨텍스트 기반. Next.js 서버 컴포넌트와의 경계 관리 복잡도 제거.
- **번들 크기**: Next.js 런타임 오버헤드 제거. Vite 빌드 결과물이 더 가볍고 빠름.
- **개발 속도**: Vite HMR이 Next.js 대비 현저히 빠름. 소규모 3화면 앱에 Next.js는 과도한 설정.
- **API 레이어**: Vercel Serverless Functions(`api/*.ts`)로 분리. Next.js Route Handlers 불필요.

### 결과

**장점:**
- 초기 로드 속도 향상 (Next.js 런타임 제거)
- 개발 환경 단순화 (Vite 설정만 관리)
- Bedrock SDK 통합 복잡도 감소

**단점/리스크:**
- SEO 없음 (앱인토스 WebView이므로 SEO 불필요 → 단점 아님)
- SSR/ISR 기반 캐싱 전략 대신 Vercel KV로 대체 (API 응답 캐싱)

**후속 조치:**
- `vite.config.ts`에 `/api/*` 프록시 설정 (로컬 개발용)
- Vercel `vercel.json`에 SPA fallback 설정 (`rewrites: [{ source: '/(.*)', destination: '/index.html' }]`)
